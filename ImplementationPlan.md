# ChessTourney Implementation Plan

## Status

- Phase: Discovery complete; planning and architecture next
- Coding status: Not started
- Last updated: 2026-04-16

## Project Summary

ChessTourney is a public web application for managing over-the-board chess tournaments, with an MVP focused on US Chess / USCF workflows and support for tournaments that are both USCF-rated and FIDE-rated when needed.

The product is intended to support multiple organizations/clubs from the start, while keeping the player experience unified and simple.

## Confirmed Requirements

### Product Scope

- The app is for over-the-board tournaments in MVP.
- The app must be designed for multi-organization use from the start.
- A single user may belong to multiple organizations.
- Initial user groups in MVP are:
  - Organization admins
  - Chief TDs
  - Staff / assistant TDs with limited permissions
  - Players
- Players should be able to register without creating an account.
- Player accounts are optional in MVP and are primarily for tournament history and autofill convenience.

### Organization And Permissions

- Permissions should follow least-privilege principles.
- Initial MVP organization-side roles are:
  - Org admin
  - Chief TD
  - Staff
- The permissions system should be designed so additional roles and permission changes can be added later.
- MFA is required for all organization-side users in MVP.
- Accessibility and password-manager compatibility are first-class requirements for authentication and form UX.

### Tournament Ownership And Rating Context

- Rated events must have exactly one affiliate.
- Unrated events may have zero affiliates.
- MVP is USCF-first.
- MVP must support a USCF event that is also FIDE rated.
- MVP does not need to support FIDE-only events.
- Non-US federation workflows are deferred unless they are low complexity after the USCF path is working.

### Tournament Formats And Structure

- MVP tournament formats must include:
  - Swiss
  - Round Robin
  - Team events
- Tournaments should support multiple sections.
- Each section should support behavior and constraints appropriate to USCF rules.
- Players register for a specific section.
- Players may register only for sections they qualify for.
- The UI should explain why a player does not qualify for a section.
- TDs may override section eligibility.
- Section eligibility should use the official posted USCF rating only.
- If player eligibility changes after registration, the app should flag the registration for TD review.
- Unrated-player section eligibility should be configurable by the TD.
- Accelerated pairings are required in MVP, using a small set of standard acceleration methods.

### Team Event Model

- Teams in MVP are captain-created from already-registered players.
- Added players must explicitly accept team membership before the roster is final.
- Players should be notified when added to a team and must be able to leave the team.
- A player may belong to only one team per event.

### Registration

- Players register by looking themselves up using name or USCF ID.
- Registration should confirm a USCF ID for rated events.
- Players without a valid USCF ID are blocked from completing registration for rated events.
- The app does not create USCF IDs for players.
- Players should be warned if membership will expire before the tournament ends and should receive a renewal link, but registration is still allowed.
- Registration constraints should be managed by the app, including section eligibility and similar tournament rules.
- TD/Staff can register walk-ins directly.
- Players should be able to cancel their own registrations.
- MVP registration statuses are registered, cancelled, checked_in, and withdrawn.
- Payment state should be modeled separately from registration state.
- Check-in is optional per event.
- Check-in in MVP is staff-managed only.

### Pairings, Results, And Operations

- Pairings are generated by the system.
- Chief TD can edit pairings.
- Staff may generate pairings.
- Staff may post pairings publicly.
- Pairings should remain hidden until posted by authorized staff or the Chief TD.
- Staff may enter results.
- MVP result entry is TD/Staff-only; there is no player self-reporting.
- Staff can manage withdrawals and byes.
- If a bye or withdrawal happens after pairings are generated, Staff can update the status and regenerate pairings.
- TDs should be able to print pairings for over-the-board event operation.
- Players should be able to view posted pairings and results online.
- TDs should be able to manage pairing changes appropriately when necessary.
- TDs should be able to enter extra rated games that are submitted with the event but do not affect event scoring/results.
- Tie-breaks should be visible to TDs in MVP.
- Public tie-break display is optional and may be deferred if that simplifies MVP.

### Visibility

- Tournament visibility is chosen per tournament by the TD.
- Default visibility should be public.
- Rosters, results, and pairings should be public in MVP.

### Payments

- MVP should support hosted online payments via a payment processor.
- MVP should also support a pay-onsite option.
- A player must either complete online payment or indicate pay-onsite to complete registration.
- A player who initially chooses pay-onsite should be able to return later and pay online.
- Payment status tracking is required.
- Waitlists are not currently in MVP scope.

### Reporting And Submission

- The app should support direct integration for USCF/FIDE submission where possible.
- Export/manual fallback should also be supported.
- Reporting validation should be implemented as much as practical.
- Tournament operations must not be blocked if the TD needs to continue running rounds.
- Final submission should be blocked when reporting validation fails.

### Auditability And Accessibility

- Sensitive actions should be auditable in MVP.
- Result corrections should keep an audit history.
- TD notes for overrides/actions should be allowed but not required.
- MVP accessibility target is WCAG 2.1 AA.
- Accessibility expectations include:
  - Keyboard-only workflows
  - Screen-reader-friendly forms
  - Strong color contrast
  - Visible focus states
  - Error summaries
  - Legible printable pairing sheets

### Notifications

- Email notifications should be opt-in.
- Passwordless authentication and passkeys are deferred to a post-MVP enhancement.

## External Verification Summary

### Verified From Official Sources

- US Chess rated events must be submitted through an active affiliate, and the submitting TD must be authorized by that affiliate.
- US Chess rating reports are submitted through MUIR and can be handled via required upload files or manual entry workflows.
- US Chess monthly supplement ratings are the official rating basis unless an event explicitly adopts a different policy.
- FIDE rating regulations require the Chief Arbiter to provide a TRF file to the federation Rating Officer, who is then responsible for uploading it to the FIDE Rating Server.

### Not Verified / Must Not Be Assumed

- No official public US Chess API documentation was found for member lookup, membership expiration verification, live rating access, or report submission.
- No public third-party FIDE API path was found for direct tournament submission beyond federation/rating-officer workflows.
- Direct machine-to-machine integrations for US Chess or FIDE should not be assumed until official documentation or direct confirmation is obtained.

### MVP Integration Stance

- US Chess support should be designed as assisted/manual-first with strong export capability.
- FIDE support should be designed as TRF/export-first with federation/rating-officer workflow awareness.
- External integrations should sit behind adapter/service boundaries so future official APIs can be added without major architectural changes.

## Next Phase: Planning And Architecture

The next phase should begin now that discovery is complete. No application code should be written until planning and architecture artifacts are reviewed and approved.

### Planning Goals

- Convert confirmed discovery outcomes into a phased implementation strategy.
- Separate MVP commitments from post-MVP enhancements.
- Identify dependencies, risks, and sequencing constraints before coding.
- Review architecture and workflow artifacts before implementation begins.

### Planning Deliverables

- MVP phase breakdown with milestones
- Deliverables, dependencies, and risks per phase
- Explicit MVP vs post-MVP scope table
- Open issues and decision checkpoints
- Recommended implementation sequence in small reviewable increments

### Architecture And Design Artifacts

The following artifacts should be produced before implementation starts:

- System context diagram
- High-level application architecture diagram
- Core domain model / entity relationship diagram
- Role and permission model
- Key workflow diagrams for:
  - Tournament creation and publishing
  - Player registration
  - Staff check-in
  - Pairing generation, review, posting, and result entry
  - Team creation and acceptance
  - Reporting/export/submission
- Integration boundary diagram covering:
  - USCF
  - FIDE
  - Payment processor
  - Email/notification provider
  - Authentication/MFA services
- Initial wireframes or low-fidelity UI flows for:
  - Public tournament discovery and registration
  - TD tournament setup
  - Section configuration and eligibility rules
  - Pairings/results management
  - Staff desk workflows

### Proposed Planning Sequence

1. Draft the architecture/design artifacts listed above.
2. Review the architecture and planning deliverables together.
3. Finalize phased implementation milestones.
4. Begin implementation only after explicit approval.

## Recommendations Recorded During Discovery

- Use multi-organization architecture with a unified public UX.
- Treat roles/permissions as extensible rather than hard-coded.
- Use hosted payment checkout rather than collecting raw card details directly.
- Keep player accounts optional in MVP.
- Prefer simple, standard acceleration options over highly custom pairing control in MVP.
- Use email/password plus TOTP MFA with recovery codes for organization-side users in MVP.
