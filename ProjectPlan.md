# ChessTourney Project Plan

## Status

- Phase: Discovery complete; planning and architecture in progress
- Coding status: Not started
- Last updated: 2026-04-16

## Purpose

`ExecutionWorkflow.md` defines how approved implementation work should be carried out.

## Project Summary

ChessTourney is a public web application for managing over-the-board chess tournaments, with an MVP focused on US Chess / USCF workflows and support for tournaments that are both USCF-rated and FIDE-rated when needed.

The product is intended to support multiple organizations/clubs from the start while keeping the player experience unified and simple.

## Current Phase

The project is currently in planning and architecture.

No application code should be written until planning artifacts and architecture/design artifacts are reviewed and approved.

## Confirmed Requirements

### Product Scope

- The app is for over-the-board tournaments in MVP.
- The app must be designed for multi-organization use from the start.
- A single user may belong to multiple organizations.
- Initial user groups in MVP are:
  - Organization admins
  - Chief TDs
  - Staff / assistant TDs with limited permissions
  - Players
- Players should be able to register without creating an account.
- Player accounts are optional in MVP and are primarily for tournament history and autofill convenience.

### Organization And Permissions

- Permissions should follow least-privilege principles.
- Initial MVP organization-side roles are:
  - Org admin
  - Chief TD
  - Staff
- The permissions system should be designed so additional roles and permission changes can be added later.
- MFA is required for all organization-side users in MVP.
- Accessibility and password-manager compatibility are first-class requirements for authentication and form UX.

### Tournament Ownership And Rating Context

- Rated events must have exactly one affiliate.
- Unrated events may have zero affiliates.
- MVP is USCF-first.
- MVP must support a USCF event that is also FIDE rated.
- MVP does not need to support FIDE-only events.
- Non-US federation workflows are deferred unless they are low complexity after the USCF path is working.

### Tournament Formats And Structure

- MVP tournament formats must include:
  - Swiss
  - Round Robin
  - Team events
- Tournaments should support multiple sections.
- Each section should support behavior and constraints appropriate to USCF rules.
- Players register for a specific section.
- Players may register only for sections they qualify for.
- The UI should explain why a player does not qualify for a section.
- TDs may override section eligibility.
- Section eligibility should use the official posted USCF rating only.
- If player eligibility changes after registration, the app should flag the registration for TD review.
- Unrated-player section eligibility should be configurable by the TD.
- Accelerated pairings are required in MVP, using a small set of standard acceleration methods.

### Team Event Model

- Teams in MVP are captain-created from already-registered players.
- Added players must explicitly accept team membership before the roster is final.
- Players should be notified when added to a team and must be able to leave the team.
- A player may belong to only one team per event.

### Registration

- Players register by looking themselves up using name or USCF ID.
- Registration should confirm a USCF ID for rated events.
- Players without a valid USCF ID are blocked from completing registration for rated events.
- The app does not create USCF IDs for players.
- Players should be warned if membership will expire before the tournament ends and should receive a renewal link, but registration is still allowed.
- Registration constraints should be managed by the app, including section eligibility and similar tournament rules.
- TD/Staff can register walk-ins directly.
- Players should be able to cancel their own registrations.
- MVP registration statuses are registered, cancelled, checked_in, and withdrawn.
- Payment state should be modeled separately from registration state.
- Check-in is optional per event.
- Check-in in MVP is staff-managed only.

### Pairings, Results, And Operations

- Pairings are generated by the system.
- Chief TD can edit pairings.
- Staff may generate pairings.
- Staff may post pairings publicly.
- Pairings should remain hidden until posted by authorized staff or the Chief TD.
- Staff may enter results.
- MVP result entry is TD/Staff-only; there is no player self-reporting.
- Staff can manage withdrawals and byes.
- If a bye or withdrawal happens after pairings are generated, Staff can update the status and regenerate pairings.
- TDs should be able to print pairings for over-the-board event operation.
- Players should be able to view posted pairings and results online.
- TDs should be able to manage pairing changes appropriately when necessary.
- TDs should be able to enter extra rated games that are submitted with the event but do not affect event scoring/results.
- Tie-breaks should be visible to TDs in MVP.
- Public tie-break display is optional and may be deferred if that simplifies MVP.

### Visibility

- Tournament visibility is chosen per tournament by the TD.
- Default visibility should be public.
- Rosters, results, and pairings should be public in MVP.

### Payments

- MVP should support hosted online payments via a payment processor.
- MVP should also support a pay-onsite option.
- A player must either complete online payment or indicate pay-onsite to complete registration.
- A player who initially chooses pay-onsite should be able to return later and pay online.
- Payment status tracking is required.
- Waitlists are not currently in MVP scope.

### Reporting And Submission

- The app should support direct integration for USCF/FIDE submission where possible.
- Export/manual fallback should also be supported.
- Reporting validation should be implemented as much as practical.
- Tournament operations must not be blocked if the TD needs to continue running rounds.
- Final submission should be blocked when reporting validation fails.

### Auditability And Accessibility

- Sensitive actions should be auditable in MVP.
- Result corrections should keep an audit history.
- TD notes for overrides/actions should be allowed but not required.
- MVP accessibility target is WCAG 2.1 AA.
- Accessibility expectations include:
  - Keyboard-only workflows
  - Screen-reader-friendly forms
  - Strong color contrast
  - Visible focus states
  - Error summaries
  - Legible printable pairing sheets

### Notifications

- Email notifications should be opt-in.
- Passwordless authentication and passkeys are deferred to a post-MVP enhancement.

## External Verification Summary

### Verified From Official Sources

- US Chess rated events must be submitted through an active affiliate, and the submitting TD must be authorized by that affiliate.
- US Chess rating reports are submitted through MUIR and can be handled via required upload files or manual entry workflows.
- US Chess monthly supplement ratings are the official rating basis unless an event explicitly adopts a different policy.
- FIDE rating regulations require the Chief Arbiter to provide a TRF file to the federation Rating Officer, who is then responsible for uploading it to the FIDE Rating Server.

### Not Verified / Must Not Be Assumed

- No official public US Chess API documentation was found for member lookup, membership expiration verification, live rating access, or report submission.
- No public third-party FIDE API path was found for direct tournament submission beyond federation/rating-officer workflows.
- Direct machine-to-machine integrations for US Chess or FIDE should not be assumed until official documentation or direct confirmation is obtained.

### MVP Integration Stance

- US Chess support should be designed as assisted/manual-first with strong export capability.
- FIDE support should be designed as TRF/export-first with federation/rating-officer workflow awareness.
- External integrations should sit behind adapter/service boundaries so future official APIs can be added without major architectural changes.

## MVP Boundary

### In MVP

- Multi-organization support
- Public tournament discovery and registration
- Optional player accounts
- Organization-side authentication with MFA
- Organization roles: Org admin, Chief TD, Staff
- Tournament creation and publication
- Multi-section tournaments
- Swiss, Round Robin, and Team events
- Section qualification rules using official posted USCF ratings
- TD override of section eligibility
- Registration, cancellation, staff walk-ins, and staff-managed check-in
- Separate payment state from registration state
- Hosted online payment plus pay-onsite flow
- Pairing generation, review, posting, and result entry
- Byes, withdrawals, and pairing regeneration workflows
- Printable pairing sheets
- Public pairings, results, and rosters
- TD-visible tie-breaks
- Extra rated games tracking
- Audit trail for sensitive actions and result corrections
- Export/manual-first reporting support for USCF and FIDE-related outputs

### Deferred / Post-MVP

- FIDE-only tournament workflows
- Non-US federation workflows unless they end up being low-complexity extensions
- Player self-reported results
- Passkeys or passwordless authentication
- Waitlists
- Advanced or highly custom pairing controls beyond standard acceleration options
- Public tie-break display if it materially complicates MVP delivery
- Fully automated federation submission unless official supported APIs are confirmed

## Recommended Delivery Strategy

Implementation should proceed in many small phases after planning approval rather than one large build-out.

Why this is the safer path:

- tournament rules, permissions, and reporting workflows have meaningful business complexity
- external federation integrations remain uncertain and should stay behind clear boundaries
- the product spans public UX, operations UX, and organization administration
- early review of architecture and workflows will reduce rework before pairing, reporting, and payment logic exist

## Phased Delivery Plan

### Phase 0: Planning And Architecture Approval

#### Goal

Produce and review the artifacts needed to start implementation safely.

#### Deliverables

- system context diagram
- application architecture diagram
- core domain model / ERD
- role and permission model
- workflow diagrams for tournament setup, registration, check-in, pairings/results, team formation, and reporting
- integration boundary diagram
- low-fidelity UI flows for public and staff-facing paths
- implementation milestone breakdown approved for coding

#### Dependencies

- confirmed discovery decisions in `ProjectPlan.md`
- review of architecture tradeoffs before implementation

#### Risks

- permissions may sprawl if role boundaries are not modeled early
- pairings and reporting workflows could force redesign if domain boundaries are vague
- unclear separation between public, player, and staff experiences could create UX debt

#### Exit Criteria

- architecture artifacts are reviewed
- implementation sequence is approved
- no unresolved planning question blocks Phase 1

### Phase 1: Foundation And Project Skeleton

#### Goal

Establish the application shell and cross-cutting foundations without implementing tournament-specific business flows yet.

#### Deliverables

- repository structure finalized
- environment and configuration strategy
- base app shell for public and authenticated areas
- authentication and MFA foundation for organization-side users
- organization membership and role primitives
- audit logging foundation
- notification abstraction and email provider boundary
- payment provider boundary
- federation/reporting adapter boundaries
- baseline accessibility standards and shared UI primitives

#### Dependencies

- approved architecture for tenancy, auth, and service boundaries
- decision on initial deployment/runtime stack if not already present in codebase

#### Risks

- overbuilding abstractions before the first real workflow
- under-modeling auditability and role checks, creating retrofits later

#### Exit Criteria

- org-side users can authenticate with MFA
- organizations, memberships, and roles are represented
- core service boundaries exist for payments, notifications, and federation/reporting support
- the project is ready for workflow-specific features

### Phase 2: Tournament Setup And Publication

#### Goal

Enable organization staff to create, configure, and publish tournaments and sections.

#### Deliverables

- tournament creation and editing flows
- affiliate assignment rules
- visibility controls
- section configuration
- section eligibility rule configuration
- check-in and payment option configuration
- public tournament listing/details page

#### Dependencies

- org and permission model from Phase 1
- domain model for tournament, section, affiliate, and eligibility concepts

#### Risks

- section and rating eligibility rules may become hard to change if captured too narrowly
- public visibility rules could conflict with draft/admin editing flows if not modeled carefully

#### Exit Criteria

- authorized staff can create draft tournaments and publish them
- public users can browse published tournaments
- sections expose eligibility and configuration rules needed for registration

### Phase 3: Registration And Player Intake

#### Goal

Support public registration and staff walk-in workflows while enforcing known eligibility and payment rules.

#### Deliverables

- player lookup flow using name or USCF ID
- registration flow for rated and unrated events
- USCF ID requirement enforcement for rated events
- membership expiration warning flow with renewal link support
- payment choice flow: online or pay-onsite
- optional player account linking/history hooks
- staff walk-in registration
- player cancellation
- TD review flags for post-registration eligibility changes

#### Dependencies

- public tournament pages from Phase 2
- membership/identity data strategy
- payment provider boundary from Phase 1

#### Risks

- player identity matching may be noisy if external lookup capability is limited
- membership verification may remain partially manual if official machine APIs are unavailable
- registration UX can become fragile if payment and eligibility states are mixed together

#### Exit Criteria

- a player can complete registration through an MVP-compliant flow
- staff can register walk-ins
- registration state and payment state are tracked separately

### Phase 4: Event Operations

#### Goal

Support in-event staffing workflows from check-in through pairings, results, and basic corrections.

#### Deliverables

- staff-managed check-in
- pairing generation flow
- pairing review and posting flow
- Chief TD pairing edit path
- result entry flow
- bye and withdrawal management
- pairing regeneration support
- printable pairing sheets
- TD-visible tie-break display
- audit history for result corrections and sensitive actions

#### Dependencies

- tournament and registration data complete enough for pairings
- approved pairing engine approach and scope for acceleration methods

#### Risks

- pairing rules are the highest-complexity tournament logic in MVP
- edit/regeneration workflows can produce data integrity problems if state transitions are under-defined
- printable outputs may uncover information-density and accessibility issues late

#### Exit Criteria

- staff can run a round operationally through the system
- posted pairings/results are visible publicly
- key corrections are auditable

### Phase 5: Team Events And Advanced Tournament Cases

#### Goal

Add the MVP-required team workflows and other event-management cases that build on the operational core.

#### Deliverables

- captain-created teams from registered players
- team invitation/acceptance flow
- leave-team flow
- one-team-per-event enforcement
- extra rated game management

#### Dependencies

- registration and event-operations foundations
- notification support for team invitations

#### Risks

- team workflows may require additional roster-state modeling
- invitation flows may complicate the optional-account/public-user model

#### Exit Criteria

- team events can be managed within MVP rules
- extra rated games are captured without affecting event scoring

### Phase 6: Reporting, Export, And Compliance Workflows

#### Goal

Support federation-facing outputs and operational validation without assuming unavailable direct APIs.

#### Deliverables

- reporting validation rules where practical
- USCF-related export/manual submission support
- FIDE-related TRF/export support
- submission readiness checks
- blocking rules for final submission when validation fails
- assisted workflow guidance for manual steps outside the application

#### Dependencies

- complete tournament and results data
- approved reporting/export format strategy

#### Risks

- official submission paths may remain partly manual
- validation coverage may differ between USCF and FIDE processes
- direct integration assumptions could create rework if added too early

#### Exit Criteria

- TDs can prepare compliant export/output artifacts
- the app supports operations continuing even when reporting issues are unresolved mid-event
- final submission readiness is clearly surfaced

### Phase 7: Validation, Hardening, And Launch Readiness

#### Goal

Prepare the MVP for reliable real-world use.

#### Deliverables

- end-to-end workflow validation
- accessibility review/remediation
- permission and audit review
- operational error handling and recovery paths
- deployment/runbook documentation
- test coverage review against MVP-critical flows

#### Dependencies

- all MVP workflows functionally complete

#### Risks

- accessibility and print workflows often need iteration late in the project
- permission regressions can hide until cross-role validation occurs

#### Exit Criteria

- MVP-critical workflows are tested
- known blockers for initial release are closed or explicitly accepted

## Cross-Phase Dependencies

- Architecture approval is the primary gate before coding.
- Pairings implementation depends on a stable domain model for tournaments, sections, registrations, rounds, and results.
- Registration work depends on a clear strategy for player lookup and membership-status verification.
- Team workflows depend on registration identity and notification capabilities.
- Reporting/export depends on complete event and result models and should remain behind integration boundaries.
- Accessibility and auditability should be built into each phase, not saved for the end.

## Key Risks To Track

- external US Chess and FIDE integration capabilities remain uncertain
- pairing and correction workflows carry the highest business-rule complexity
- optional player accounts plus public registration create identity-linking edge cases
- role boundaries may drift unless permission checks are centralized early
- printable operational materials may have requirements not obvious from screen flows alone

## Decision Checkpoints Before Coding

- approve the architecture and workflow artifacts
- confirm the initial technical stack if a codebase foundation does not already exist
- confirm the first implementation milestone after planning approval
- confirm whether any non-MVP reporting automation should be intentionally deferred even if technically feasible

## Recommended First Implementation Milestones After Approval

1. Foundation/auth/organization setup
2. Tournament setup and publication
3. Registration and payment selection
4. Event operations core

This sequence delivers usable surface area early while deferring the most integration-sensitive reporting work until the core tournament model has proven out.

## Recommendations Recorded During Discovery

- Use multi-organization architecture with a unified public UX.
- Treat roles/permissions as extensible rather than hard-coded.
- Use hosted payment checkout rather than collecting raw card details directly.
- Keep player accounts optional in MVP.
- Prefer simple, standard acceleration options over highly custom pairing control in MVP.
- Use email/password plus TOTP MFA with recovery codes for organization-side users in MVP.
